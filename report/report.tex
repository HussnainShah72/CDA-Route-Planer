\section{Introduction}
The CDA Bus Route Analysis project applies advanced process mining techniques to the public transit network of the Islamabad Capital Territory. By utilizing a highly modular Python architecture, the system translates raw transit schedules into actionable event logs, performs process discovery, and identifies infrastructural bottlenecks. Furthermore, the system integrates a Large Language Model (LLM) powered by Groq to serve as an interactive, data-grounded Agentic AI trip planner. The entire system is encapsulated within a custom-designed, premium Streamlit graphical user interface (GUI).

\section{System Architecture \& Workflow}
The project is built with a strictly modular architecture to separate data processing, analytical logic, and user interface components.

\subsection{Project File Structure}
The following structure ensures scalability and maintainability:
\begin{itemize}
    \item \textbf{Root Directory:}
    \begin{itemize}
        \item \texttt{report.pdf}: The academic documentation source.
        \item \texttt{README.md}: Setup and deployment instructions.
        \item \texttt{requirements.txt}: Python environment dependencies.
    \end{itemize}
    \item \textbf{\texttt{data/}}: Data storage and validation layer.
    \begin{itemize}
        \item \texttt{routes.csv}: Cleaned, structured transit dataset.
        \item \texttt{cda\_bus\_routes.xes}: The formalized process mining event log.
        \item \texttt{personal\_routes.csv}: Commute definitions for group members.
        \item \texttt{verify\_xes.py}: Automated validation script for XES integrity.
        \item \texttt{raw\_pdfs/}: Local cache of original CDA transit schedules.
    \end{itemize}
    \item \textbf{\texttt{code/}}: Application and Logic layer.
    \begin{itemize}
        \item \texttt{app.py}: The primary Streamlit GUI and controller.
        \item \texttt{run\_pipeline.py}: CLI tool for end-to-end data processing.
        \item \textbf{\texttt{pms\_project/}}: Core library modules.
        \begin{itemize}
            \item \texttt{process\_mining.py}: Trace construction and DOT logic.
            \item \texttt{analytics.py}: Throughput and Bottleneck math.
            \item \texttt{rag\_agent.py}: LangChain / Groq AI integration.
            \item \texttt{personal\_maps.py}: Orientation and shortest-path logic.
            \item \texttt{commute\_case.py}: Manual trace builders for bonus tasks.
            \item \texttt{extraction.py}: PDF parsing and data cleaning logic.
            \item \texttt{config.py}: Global paths and shared constants.
        \end{itemize}
    \end{itemize}
\end{itemize}

\subsection{Operational Workflow}
The system operates through a sequential data pipeline as shown in the workflow below:
\begin{enumerate}
    \item \textbf{Data Ingestion:} Raw schedules are loaded via Pandas and cleaned for timestamp consistency.
    \item \textbf{Trace Construction:} Individual bus trips are grouped into "Traces" and exported to the IEEE XES standard format.
    \item \textbf{Process Discovery:} Directly-follows relationships are calculated to build a Transition Matrix.
    \item \textbf{Performance Layer:} The matrix is enriched with timing data to calculate throughput and flag delays.
    \item \textbf{Interactive Layer:} The Streamlit GUI renders the analytics while the RAG Agent provides a natural language interface to the same underlying graph.
\end{enumerate}

\section{Task 1: Data Extraction}
\subsection{Objective}
Extract, clean, and structure the raw CDA bus transit schedules into a formalized dataset suitable for advanced process mining and algorithmic trace log construction.

\subsection{Implementation Details}
The \texttt{task1\_data\_extraction} module was developed to parse the raw transit tables provided. The raw data contained route designations, stop locations, and corresponding temporal data. 
This data was rigorously cleaned and transformed into a standardized Pandas DataFrame (\texttt{routes.csv}) containing structured columns: \texttt{route\_id}, \texttt{stop\_name}, \texttt{arrival\_time}, and \texttt{departure\_time}. Data types were strictly cast, and sequential ordering was enforced to guarantee temporal consistency for downstream case inference.

\section{Task 2: Trace Log Construction}
\subsection{Objective}
Construct a process mining event log from the raw bus dataset by exporting it as an XES-compliant file, validating it for standard academic and industrial process mining workflows.

\subsection{Implementation Details}
The transit data was parsed into a structured format where individual bus trips were inferred as unique "cases" by grouping consecutive stops along specific \texttt{route\_id}s. 
The \texttt{xes\_builder.py} module was implemented to iterate over these grouped dataframes, generating standard XES XML trees. 
\begin{itemize}
    \item \textbf{Metadata:} Global extensions (Concept, Time, Lifecycle) were injected at the root level, alongside \texttt{xes.version="1.0"}.
    \item \textbf{Event Tracking:} Every stop visit was recorded as an event, capturing \texttt{stop\_name} as the \texttt{concept:name} and properly formatting the departure and arrival times into ISO 8601 strings with timezone localization.
\end{itemize}

\subsection{Tool Validation}
The generated \texttt{cda\_event\_log.xes} file was verified by importing it into an external Process Mining Tool to ensure structural integrity and correct trace mapping.

\section{Task 3: Process Discovery \& Interactive GUI}
\subsection{Objective}
Discover a formal process map from the event logs and present it in an interactive GUI supporting dynamic filtering and rich edge annotations.

\subsection{Implementation Details}
A premium, dark-themed (Charcoal and Gold) Streamlit dashboard was developed. Process mapping was implemented without relying solely on black-box external tools; instead, a custom \texttt{transition\_table} was calculated using pandas DataFrame shifting to identify directly-follows relationships between transit stops.

\begin{itemize}
    \item \textbf{Graphviz Rendering:} The \texttt{process\_mining.py} module translates the transition matrices into Graphviz DOT syntax. The graph background is set to a high-contrast cream (\texttt{\#E8DDB4}) with all non-bottleneck edges rendered as solid black for maximum visibility.
    \item \textbf{Route Filtering:} The sidebar provides a dynamic select box. When a specific route (e.g., FR-01) is chosen, the underlying dataframe is filtered, instantly re-rendering a highly focused sub-graph. Selecting "All Routes" displays the interconnected macro-network.
    \item \textbf{Strict Label Formatting:} In accordance with the assignment requirements, every directed edge is explicitly annotated using the format: \textbf{\texttt{Source $\to$ Target = Duration (cases: n)}}. For example, \textit{PAEC $\to$ IMC = 1 min 45 sec (cases: 5)}.
\end{itemize}

\section{Task 4: Performance \& Bottleneck Analytics}
\subsection{Objective}
Enrich the GUI with statistical timing information and automatically highlight transit bottlenecks based on empirical data.

\subsection{Implementation Details}
Performance analytics were deeply integrated into the analytical pipeline:
\begin{itemize}
    \item \textbf{Throughput Metrics:} The total end-to-end duration for every individual case was calculated ($T_{total} = t_{last\_departure} - t_{first\_arrival}$). The globally computed Average, Minimum, and Maximum throughput times are displayed natively via highly visible metric cards at the top of the dashboard.
    \item \textbf{Bottleneck Detection:} A dynamic threshold slider allows stakeholders to define what constitutes a delay (e.g., transitions taking longer than 5 minutes). The system loops over the transition matrix; any edge whose \texttt{avg\_duration\_seconds} exceeds this threshold is highlighted in a thick red (\texttt{\#C62828}) stroke on the Process Graph.
    \item \textbf{Bottleneck Summary:} A dedicated GUI panel lists the top slowest transitions, sorted in descending order, to immediately draw attention to the most severe infrastructural delays.
\end{itemize}

\section{Task 5: Agentic AI Trip Planner}
\subsection{Objective}
Integrate a conversational AI agent into the GUI that answers natural-language trip-planning queries grounded in the discovered CDA route data, strictly avoiding hallucinations.

\subsection{Implementation Details}
To bridge the gap between process mining and end-user utility, a "Grounded AI Trip Planner" was embedded into the GUI. 
\begin{itemize}
    \item \textbf{Agent Architecture:} The system utilizes a LangChain-powered agent backed by Groq's LLaMA-3.3-70b model. It uses a Retrieval-Augmented Generation (RAG) pattern to fetch real-time scheduling data.
    \item \textbf{Data Grounding:} Instead of relying on the LLM's parametric memory, the agent utilizes a custom \texttt{GroundedRouteAgent} that performs graph searches over the actual \texttt{routes.csv}.
    \item \textbf{Premium Chat UI:} The planner is enclosed in a styled \texttt{chat-container} within the GUI, featuring an inline input field and visually structured responses (markdown headers, emojis, and code-block path visualizations) for a premium user experience.
\end{itemize}

\section{Bonus Task: Personal Route Maps}
Going beyond the assignment parameters, a custom \textbf{Personal Route Maps} feature was implemented for every team member. 
\begin{itemize}
    \item \textbf{Member-Specific Visualization:} Each member (Hussnain, Uzair, Faez, and Hashir) has their specific daily commute visualized as a separate, isolated process map.
    \item \textbf{Interactive Trace Builder:} An interactive "Custom Route Builder" was added to the GUI, allowing any user to select arbitrary origin and destination stops. The backend instantly computes the shortest commuting path and generates a personalized route map with transition timings.
    \item \textbf{Aesthetic Consistency:} Personal maps follow the same high-contrast labeling requirements as the main process map, including explicit duration and route identification on every edge.
\end{itemize}
